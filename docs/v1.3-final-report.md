## Что добавлено

### Health endpoint

`GET /api/pipeline/health` (owner|pm) — single-call snapshot для мониторинга:

```json
{
  "status": "ok",
  "counts_by_status": {"running": 2, "done": 47, "failed": 1, ...},
  "recent_24h": {"created_24h": 5, "completed_24h": 4, "failed_24h": 1},
  "total_tokens_used": 1250000,
  "workspace_size_mb": 23.4,
  "db_size_mb": 12.1,
  "tmux_available": true,
  "claude_api_key": "configured" | "disabled"
}
```

Формат идеален для Healthchecks.io / Uptime Kuma / любого external monitor.

### Search в pipeline.html

Search input в filters-row. Client-side debounced (200ms) фильтр по:
- `title`
- `raw_idea`
- `project_type`
- `id`

Работает поверх already-loaded list (без extra API calls), мгновенный отклик. Когда runs накопится >50 — без поиска неудобно.

Также filter-status добавил недостающие опции (paused_user, aborted).

### Audit log

`pipeline_audit_log` table — отдельная сущность для compliance + debug:

```sql
CREATE TABLE pipeline_audit_log (
    id, run_id, action, actor, actor_id, source,
    from_status, to_status, details_json, created_at
);
```

Записывается на каждое admin-action:
- `create` — POST /runs (HTTP)
- `pause` / `resume` / `abort` / `approve` — both HTTP and Telegram

Поля `actor` (`owner`/`telegram_owner`/`system`) и `source` (`http`/`telegram`/`system`) показывают **кто и откуда** сделал.

`GET /api/pipeline/audit-log?run_id=&action=&limit=&offset=` (owner-only) — для read.

`pipeline/audit.py` — общий модуль с `log_action()` и `list_audit_log()`. Best-effort: если запись audit упадёт, основное действие НЕ блокируется (try/except внутри log_action).
