# Аудит проекта AI Delivery HQ

**Дата:** 2026-05-15
**Цель аудита:** карта существующего кода для планирования замены текущей AI Команды на multi-agent pipeline автономной разработки клиентских проектов через Claude Code.
**Режим:** read-only. Никакие файлы кроме этого отчёта не созданы и не изменены.
**Источник истины:** прямой обход кода через 4 параллельных Explore-агента + ручная сверка ключевых файлов.

---

## TL;DR (для ленивого чтения)

Проект — операционное ядро AI-агентства (FastAPI + SQLite + 23 AI-агента + Telegram-бот + панель HQ), задеплоенное на VPS `89.22.235.144`. Около **170 эндпоинтов**, **46 таблиц**, **18 HTML-страниц** панели, всё на ванильном JS без сборщика.

**Что критично перед началом интеграции:**

1. **Нет git вообще.** Ни в `AI_Delivery_Team/`, ни в `ai_agency/`, ни выше. Никакой истории, никаких ревью, никакой возможности откатиться. Это первое что нужно поправить, ещё до начала переноса AI Команды.
2. **`.env.example` содержит реальные секреты.** Не плейсхолдеры — настоящий ANTHROPIC_API_KEY, токен Telegram-бота и пароль админки `Vfhnsyjdf1953!` (раскладка для «Мартынова1953!»). На проде в `.env` другой пароль — `Admin2024` — тоже слабый.
3. **AI Команда структурно работоспособна.** Грепы по `TODO/FIXME/HACK/NotImplementedError/заглушка/временно` в её ядре дают чисто. Если ощущение что «сломана» — это про ROI, UX или скрытые runtime-баги, а не про сломанный код.
4. **Делать новый «AI Pipeline» как отдельный модуль рядом, не вместо.** Таблицы `delivery_*` готовы под workflow клиентских проектов. Старая AI Команда тесно сплетена с Telegram-ботом — резкое удаление сломает бот.

### 1.2 Точки входа

- **`python ai_agency/main.py`** — единственная точка входа в прод. Запускает FastAPI (uvicorn) + Telegram-бот + scheduler в одном asyncio event loop.
- **`WEB_ONLY=true python main.py`** — режим без бота и планировщика (для локальной разработки, чтобы не ломать polling прода).
- **systemd-юнит `ai-agency`** на сервере дёргает то же самое.
- **Shell-агенты** (`scripts/01..12_*.sh`) — отдельная сущность, не часть HQ. Запускают Claude Code сессии с разными SYSTEM_PROMPT.md из `01_AGENTS/`. Похоже на устаревший прототип до того как появился `ai_agency/`.

## 2. База данных

### 2.1 Расположение и бэкапы

- **Главная БД:** `ai_agency/agency.db` (SQLite, 778 КБ).
- **Тестовая:** `ai_agency/_test_p6.db` (229 КБ) — для `_release_gate_test.py`/`test_full.py`.
- **Бэкапы:** `ai_agency/backups/agency_YYYYMMDD_HHMMSS_startup.db`. Делаются при каждом старте `init_db()` + по `POST /api/backup`. На момент аудита — 7 файлов, последний от 29.04.2026. **Ротация: нет.** Папка просто растёт.

### 2.2 Подход к миграциям

**Без Alembic.** Всё в `init_db()` (database.py:66) последовательно:

```
init_db()
  → CREATE TABLE IF NOT EXISTS … (46 таблиц)
  → CREATE INDEX IF NOT EXISTS … (39 индексов)
  → _add_column_if_missing(db, table, col, type) для каждой новой колонки
  → _migrate_clients_service_type_drugoe()
  → _migrate_v3() — главная миграция (46+ колонок, 14+ новых таблиц)
  → _migrate_remove_check_constraints() — снимает CHECK для гибкости статусов
  → _migrate_fix_project_tasks_fk() / _fix_projects_fk() / _fix_payments_fk()
  → _migrate_drop_legacy_tables() — чистит *_old, *_broken
  → _migrate_drop_delivery_project_members() — удаляет умершую таблицу
  → _migrate_executors_level()
  → _migrate_delivery_templates_expand()
  → _seed_owner_user() — ensure user id=1
  → _seed_delivery_templates() — 8 шаблонов (Лендинг, Многостраничный, Telegram-бот, AI-ассистент, n8n, …)
```

Идемпотентность через `PRAGMA table_info` + try/except. `PRAGMA foreign_keys=OFF` + `PRAGMA legacy_alter_table=ON` оборачивают структурные изменения.

### 2.4 Конвенции

- ID — `INTEGER PRIMARY KEY AUTOINCREMENT`.
- Времена — **смешано**: старые таблицы `TEXT DEFAULT (datetime('now'))`, новые `TIMESTAMP DEFAULT CURRENT_TIMESTAMP`. Эквивалентно для SQLite, но визуально путает.
- Soft delete — только `clients.is_archived`. У остальных — через статус или нет вообще.
- JSON — храним как `TEXT '[]'` / `'{}'`, парсинг на стороне Python.
- CHECK constraints удалены миграцией `_migrate_remove_check_constraints` — статусы валидируются на уровне приложения.

### 2.5 Запахи схемы (что мешает)

1. **`status` vs `status_ru`** в `clients` — дублирование, один источник истины не определён.
2. **Денормализованные revenue-поля** (`our_percent`, `our_amount`) могут разъехаться — нет триггеров.
3. **`reminders.related_id`** — нетипизированный FK, тип определяется полем `type`. Без референциальной целостности.
4. **`hq_agent_messages` без `task_id`** — история чата не привязана к задаче.
5. **`tasks` и `tasks_v2`** — две параллельные системы задач, не одна с миграцией. `tasks_v2.ai_task_id` пытается мостить, но это костыль.
6. **`hq_users` и `executors`** — почти дубль (поля name/role/telegram/email/github_username/specialization повторяются). `executors.user_id` существует, но в чём смысл двух таблиц — неочевидно.
7. **`knowledge_base.chunks`** — JSON в TEXT, без индексов. Полнотекстовый поиск не настроен.
8. **Нет таблицы версий схемы** — не понять «на какой миграции мы стоим».

## 3. Backend API

### 3.1 Архитектура

- **Один FastAPI app** в `api.py` (`app = FastAPI(...)`). `hq_v3_api.py` импортируется как модуль и **навешивает свои `@app.get/post/...` на тот же `app`** через scope-замыкание. APIRouter не используется.
- **Статика монтируется** на старте: `/hq` → `static/hq/`, `/static` → `static/`, `/service` → `static/site/`.
- **CORS** включён, есть middleware для редиректа `/hq` → `/hq/` (trailing slash).
- **Бизнес-логика — толстая в роутах** (10–60 строк на эндпоинт). Сервис-слоя нет. БД-функции в `database.py` тонкие, обычно 1–2 SQL.
- **Долгие задачи** запускаются через `asyncio.create_task()` (api.py:451) и не блокируют ответ.

### 3.3 Авторизация (двойная)

**Legacy:**
- Заголовок `X-Admin-Password` или query-параметр `password=...` сверяется с `ADMIN_PASSWORD` из `.env`.
- `verify_password()` (api.py:136) — auto-promote до `role: "owner"`.
- ⚠ Передача пароля в query попадает в логи uvicorn.

**Новая (session-based):**
- `POST /api/auth/login` → `secrets.token_hex(32)` → хранится в **in-memory dict `_sessions`** (hq_v3_api.py:87).
- TTL: 7 дней. Заголовок `X-Auth-Token`.
- Пароль хешируется SHA256 (без bcrypt/argon2).
- ⚠ **Все сессии теряются при рестарте сервиса.**

**CSRF:** не реализован. Ставка на SameSite/CORS.
**Rate limiting:** нет вообще. `POST /api/auth/login` можно брутить.

### 3.4 WebSockets

Один — `/ws/task/{task_id}` (api.py:737). Подписывается на `TaskProgress` orchestrator'а, стримит шаги выполнения. Используется на `team.html` для прогресс-бара.

### 3.5 Подозрительное

- **In-memory сессии** — рестарт = всех разлогинивает. Для долгих Claude-Code пайплайнов это будет критично.
- **Пароль в query** (`?password=...`) — в логах.
- **`agent_name` из request body не валидируется** против списка известных агентов (api.py:424).
- **Cascade delete без подтверждения** — `DELETE /api/clients/{id}` дёргает каскад.

### 5.2 Pipeline

```
POST /api/task (api.py:424)
   → orchestrator.run_task(message)
       → ChiefOfStaff.think() → JSON-план (name, brief, priority, assigned_department)
       → _resolve_task_mode() → lite/standard/full
       → _should_route_client_solutions() — keyword match
       │
       ├─ если AI Solutions: run_client_analysis()
       │     ClientCEO → AIStrategist → optional Crisis/PM → синтез
       │
       └─ иначе: для каждого assigned_department:
```

### 5.6 Что именно «сломано» (если по коду)

**По коду — ничего критичного.** Грепы по `TODO/FIXME/XXX/HACK/NotImplementedError/заглушка/временно` в `orchestrator.py`, `agents/{base,team,context,request_context}.py`, `agency_context_loader.py` дают **ноль попаданий**. Все три режима (lite/standard/full) имеют полные code paths. Все 23 агента зарегистрированы в `_registry` и доступны через `team.list_agents()`.

**Но есть слабости, которые могут восприниматься как «сломано»:**

1. **Tool use не реализован.** Агенты — это голые `messages.create()` без tools, без structured output, без streaming от Anthropic. Просто текст в ответ. Для автономной разработки клиентских проектов это блокер — Claude Code наоборот вокруг tools построен.
2. **Prompt caching не используется.** Системный промпт перестраивается на каждый вызов → лишние токены и латентность.
3. **Нет ретраев.** Любая ошибка API возвращается как failed `AgentResponse`. Под нагрузкой/rate-limit поведение хрупкое.
4. **Нет батчей и async-стриминга.** Параллелизм на уровне `asyncio.gather()` по департаментам, но внутри департамента — последовательно.
5. **Контекст агентства весит.** 60с кэш + 40 элементов knowledge_base × 2000 chars каждый = в худшем случае ~80KB на каждый запрос. Это не tool use, это просто простыня в system prompt.
6. **Нет связи `hq_agent_messages` → `tasks`.** История чата с агентом отвязана от задач, которые из этого чата родились.
7. **Telegram-бот напрямую дёргает orchestrator** (`telegram_bot.py` → `Orchestrator.run_task`). Если выкинуть orchestrator — бот ломается.

Если ощущение «сломано» исходит из «не вижу выхлопа от 23 агентов» — это вопрос про ROI/UX, не про код. Код работоспособный, но это **prompt-only система**, которая выдаёт текст. Для автономной разработки кода — нужна другая парадигма.

### 9.2 Что сломано / недоработано

## 10. Резюме для планирования AI Pipeline

**Что радует:**
- Готовая инфра для трекинга проектов (`delivery_*` таблицы) — фактически бесплатный фундамент для pipeline runs.
- `executors` + `hq_users.github_username` — связь людей и github.
- WebSocket-стриминг прогресса уже работает.
- Auth/CRM/проекты/клиенты/документы — всё есть и стабильно.
- Документация (`ai_agency/CLAUDE.md`) высокого качества.

**Что блокирует или создаёт риск:**
- Нет git → не начнём ничего.
- Утечка секретов → переустановить ключи.
- Старый orchestrator привязан к Telegram-боту → не выпиливать резко.
- SQLite + параллельные subagent'ы → возможны блокировки.
- `delivery.html` может быть уже сломан в проде (не подтверждено).

**Стратегия (high-level, для брейншторма):**
1. Добавить git, отозвать ключи, перенести сессии в DB.
2. Создать `pipeline/` модуль рядом с `agents/`.
3. Использовать `delivery_*` таблицы как backbone (один pipeline run = один `delivery_project`, фазы = `delivery_stages`, шаги = `delivery_tasks`).
4. Старую AI Команду оставить живой ещё 1-2 итерации — пока Telegram-бот не перепишем.
5. Постепенно гасить старые агенты и блоки UI, заменяя на новые.

Готовность кодовой базы к замене AI Команды: **6/10**. Архитектура хорошая, но нужно сначала закрыть инфра-долг (git, секреты, сессии), а уже потом строить новое.
